# DBスキーマ変更の注意点

**最終更新**: YYYY-MM-DD

データベーススキーマを変更する際の注意点とコツです。

---

## カラム名の変更

### 手順
1. 新しいカラムを追加
2. データ移行（旧カラム → 新カラム）
3. アプリケーションコードを新カラムに切り替え
4. 旧カラムを削除

### 注意点
- いきなりRENAMEしない（アプリが壊れる）
- 段階的に移行する
- 移行期間中は両方のカラムを維持

---

## カラムの削除

### 手順
1. アプリケーションコードから参照を削除
2. 本番デプロイして問題ないことを確認
3. カラムを削除

### 注意点
- 参照箇所を全て確認してから削除
- 外部キー制約がある場合は先に制約を削除
- 削除前にバックアップを取る

---

## テーブル名の変更

### 手順
1. 新しいテーブルを作成
2. データを移行
3. アプリケーションコードを切り替え
4. 旧テーブルを削除

### 注意点
- VIEWを使った段階的移行も検討
- 外部キー参照がある場合は慎重に

---

## 型の変更

### 安全な変更
- VARCHAR(50) → VARCHAR(100)（拡大）
- INT → BIGINT（拡大）

### 危険な変更
- VARCHAR(100) → VARCHAR(50)（縮小、データ切り捨て）
- VARCHAR → INT（変換失敗の可能性）

### 注意点
- 既存データが新しい型に収まるか確認
- 縮小変更は事前にデータ検証必須

---

## 共通の注意点

1. **本番環境での直接変更は避ける**
   - 開発環境で十分にテスト
   - マイグレーションスクリプトを用意

2. **バックアップ必須**
   - 変更前に必ずバックアップ

3. **外部キー制約の確認**
   - 変更対象のカラム/テーブルを参照している箇所を確認

4. **ダウンタイムの考慮**
   - 大きな変更はメンテナンス時間を設ける
