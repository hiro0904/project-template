# Presentations（プレゼン資料管理）

**最終更新**: 2026-04-23

ミーティング・発表用のプレゼン資料を管理するディレクトリ。
ローデータの収集から原稿作成、スライド自動生成までのワークフローを提供する。

---

## ディレクトリ構成

```
presentations/
├── README.md                  # 本ファイル（運用ルール・制作フロー）
└── mtg-XXXX-テーマ名/         # ミーティング・発表単位
    ├── draft.md               # 骨格 + 原稿
    ├── slide.md               # スライド構成書（/gen-slide で生成）
    └── work/
        ├── raw/               # ローデータ保管庫（生データのみ）
        └── (中間整理ファイル)   # raw/を分析・整理したもの
```

## 命名規則

```
mtg-0001-週次報告, mtg-0002-Q2振り返り, mtg-0003-新機能提案 ...
```

4桁番号 + テーマ名。番号は通し番号。

---

## 全体フロー

```
Phase 1: ヒアリング    → work/raw/hearing-XX.md に音声入力を記録
Phase 2: リサーチ      → work/raw/research-XX.md に調査結果を蓄積
Phase 3: 中間整理      → work/ 直下に分析・絞り込みファイルを作成
Phase 4: 骨格作成      → draft.md の前半に構成案を作成
Phase 5: 本文作成      → draft.md の後半に発表原稿を執筆
Phase 6: スライド生成   → /gen-slide でスライド構成書 + PPTX自動生成
Phase 7: 発表          → PowerPointで確認・微調整して発表
```

---

## Phase 1: ヒアリング

### 何をやるか

ユーザーが「話したいこと」「考えていること」を音声入力で喋り、Claude Codeがそのまま記録する。

### やり方

1. ユーザーが音声入力（Aqua Voice等）で発話する
2. Claude Code が受け取った内容を **整形せず** そのまま `work/raw/hearing-XX.md` に保存する
3. 1回のセッション（まとまった発話）につき1ファイル作成

### ファイルの書き方

```markdown
# ヒアリング記録 XX

**日付**: YYYY-MM-DD

---

## 発話メモ（生データ）

（ユーザーの発話をそのまま記載。整形・要約しない）
```

### ポイント

- **整形しない**。音声入力の誤変換もそのまま残す
- 1つの発話で複数テーマが出てきたらそのまま書く（分割はPhase 3でやる）
- 同じ日に複数回のヒアリングがあれば `hearing-01.md`, `hearing-02.md` と分ける
- ヒアリング中に「これ調べて」という指示が出たら、その場でメモしておいてPhase 2で対応

---

## Phase 2: リサーチ

### 何をやるか

ヒアリングで出てきた疑問や検討事項について調査し、結果を記録する。

### やり方

**パターンA: Claude Codeが直接調べる**
1. ヒアリング内容から調べるべきことを洗い出す
2. Web検索やコード検索で調査
3. 結果を `work/raw/research-XX-テーマ.md` に保存

**パターンB: プロンプトを用意して外部で調べる**
1. 調査用プロンプトを `work/raw/research-prompts.md` にまとめる
2. ユーザーがClaude Web等で実行
3. 結果をコピーして `work/raw/research-XX-テーマ.md` に貼り付け

### ファイルの書き方

**プロンプト集の例:**
```markdown
# リサーチ用プロンプト集

**日付**: YYYY-MM-DD

> Claude Webで使うプロンプト。結果は別ファイルとしてwork/raw/に貼り付ける。

---

## 自分の条件（全プロンプト共通の前提）

以下をプロンプトの冒頭に貼って使う：

（共通の前提条件を記載）

---

## プロンプト①: （調査テーマ）

→ 結果ファイル: `research-01-xxx.md`

（プロンプト本文）
```

**結果ファイルの例:**
```markdown
# （調査テーマ）

（調査結果をそのまま貼り付け。出典やソースがあれば記載）
```

### ポイント

- プロンプト集と結果ファイルは分ける（プロンプトの再利用性を保つ）
- 結果ファイル名にはテーマを含める（`research-01-jobs.md`, `research-02-companies.md` 等）
- 調査結果は加工せずそのまま保存。分析はPhase 3で行う

---

## Phase 3: 中間整理

### 何をやるか

raw/ に溜まったヒアリングとリサーチの情報を分析・整理し、draft.mdに進むための土台を作る。

### やり方

1. raw/ のファイルを読み返す
2. テーマごとに情報を整理した分析ファイルを `work/` 直下に作成
3. 候補の絞り込み、比較表、方向性の検討などを行う

### ファイルの書き方

```markdown
# （分析テーマ）

**最終更新**: YYYY-MM-DD

> リサーチ結果（`work/raw/research-XX.md`）を元に、（目的）。

---

## 前提条件（ヒアリングから）

- ...

---

## （分析内容）

（表、比較、絞り込みなど）
```

### ポイント

- raw/ は生データ専用。分析・整理は `work/` 直下に置く
- raw/ のファイルは削除・上書きしない（生データは資産として残す）
- 中間整理は必須ではない。ヒアリングとリサーチだけで骨格が作れるならスキップしてよい

---

## Phase 4-5: draft.md（骨格 + 原稿）

### 何をやるか

Phase 1-3 で溜めた情報をもとに、発表の構成と原稿を作成する。

### 構成

draft.md は **骨格** と **本文** を1ファイルにまとめる。

```markdown
# [発表テーマ]

**最終更新**: YYYY-MM-DD

> このファイルには **骨格** と **最終原稿** を両方置く。
> ローデータ（生の発話・思考メモ）は `work/raw/` に保管。

---

## 伝えたいこと

- （この発表で一番伝えたいことを箇条書きで整理）
- （聞き手に何を持ち帰ってほしいか）

---

## 骨格

1. **セクション名** — 要点の説明
2. **セクション名** — 要点の説明
3. ...

---

## 本文

### 1. セクション名

（実際に発表で話す内容をそのまま書く）

### 2. セクション名

（本文）

...
```

### 骨格の作り方

1. raw/ のヒアリング内容から「伝えたいこと」を抽出
2. セクション構成を決める（導入→本題→まとめ の流れ）
3. 各セクションの要点を1-2行で記述
4. ユーザーと構成を確認してから本文に進む

### 本文の作り方

1. 骨格のセクション順に、実際に話す内容を書く
2. raw/ のヒアリング内容やリサーチ結果を参照しながら肉付け
3. 話し言葉で書く（発表原稿なので、書き言葉にしすぎない）
4. 1セクションごとに書いてユーザー確認 → 修正 を繰り返す

---

## Phase 6: スライド生成

### 何をやるか

draft.md の原稿をもとに、PowerPoint スライドを自動生成する。

### やり方

```
/gen-slide presentations/mtg-XXXX-テーマ名/
```

Claude Code が以下を実行：
1. `slide.md`（スライド構成書）を作成 → ユーザー確認
2. `tools/pptx/scripts/mtg-XXXX.js`（生成スクリプト）を作成
3. スクリプトを実行して PPTX を出力

詳細は `docs/presentation-system.md` を参照。

## work/raw/ — ローデータ保管庫

全てのインプットをここに集約する。情報を落とさないための置き場所。

### 入るもの

| 種類 | ファイル名の例 | 内容 |
|---|---|---|
| **ヒアリング** | `hearing-01.md` | ユーザーの発話記録（音声入力そのまま） |
| **リサーチプロンプト** | `research-prompts.md` | 調査用プロンプトの集約 |
| **リサーチ結果** | `research-01-jobs.md` | 調査結果（そのまま貼り付け） |
| **メモ** | `ideas.md` | 思いつき、アイデアの種 |
| **外部情報** | `chatgpt-xxx.md` | 外部ツールの出力結果 |

### 運用ルール

- ファイル形式は **.md に統一**
- ファイル名は内容が分かる単位で自由に分ける
- 音声入力の生テキストは**そのまま貼ってOK**（整形しない）
- 検索結果やツール出力も**そのまま貼ってOK**
- **ローデータは削除・上書きしない**（後から参照する資産として残す）
