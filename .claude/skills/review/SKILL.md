# コードレビュー実施

引数 `$ARGUMENTS` からIssue番号を取得。

## 手順

1. **レビュー対象の特定**: 作業ログ `dev-logs/` とissueファイル `issues/issue{番号}-*.md` から変更内容・仕様を把握
2. **変更ファイルを全て読む**: 作業ログの変更ファイル一覧に記載されたファイル（なければ `git diff` で特定）
3. **レビュー観点でチェック**: `reviews/README.md` の「レビュー観点」に従って精査
4. **レビュー記録作成**: `reviews/issue{番号}-review.md` を `reviews/README.md` のテンプレートに従って作成
5. **完了報告**: レビュー結果サマリーと対応要否を報告

## ドキュメント整合性チェックリスト（必須）

レビュー時に必ず以下を確認し、問題があれば指摘すること：

- [ ] `issues/issue{番号}-xxx.md` が作成されているか
- [ ] `backlog.md` から完了した項目が削除されているか
- [ ] `backlog.md` に「実装完了」等の履歴記述が残っていないか（未対応タスクのみ記載。完了履歴は closed-issues.md で管理）
- [ ] `closed-issues.md` に追加されているか
- [ ] 関連README更新済み（該当する場合）
- [ ] 関連する仕様書が更新されているか（該当する場合）

## 参照ドキュメント

- `reviews/README.md` — レビュー観点・指摘対応ルール・テンプレート

## 注意

- 日付は必ず `date +%Y-%m-%d` で取得する
- 実際のコードを読んでレビューする（想像で指摘しない）
- 完了報告時、各指摘に重要度（致命的/重要/軽微）を明記し、コミット前に必要な対応を明示する
- レビュー指摘は「その場で修正」「backlog追記」「ユーザー判断で棄却」のいずれかで対応してからコミットに進む
