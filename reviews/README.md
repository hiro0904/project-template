# レビュー運用

**最終更新**: 2026-03-14

Issue単位でのコードレビュー記録を管理するディレクトリ。

## ファイル構成

```
reviews/
├── README.md              # 本ファイル（レビュー手順）
└── issue{番号}-review.md  # 各Issueのレビュー記録
```

**例**: `issue1-review.md`, `issue2-review.md`
**複数Issue対応の場合**: `issue1-2-review.md`

## レビュータイミング

```
1. 作業完了
     ↓
2. 作業ログ作成（dev-logs/）
     ↓
3. Claude Codeにレビュー依頼
     ↓
4. レビュー記録作成（reviews/）
     ↓
5. レビュー指摘対応（全指摘を解決してからコミット）
   - その場で修正 / ユーザー判断で棄却
     ↓
6. コミット・push
```

## レビュー依頼の仕方

```
「Issue #1 の作業が完了したのでレビューして」
「作業ログ作成後、レビューもお願い」
```

## レビュー観点

### 1. 保守性・コード品質
- 可読性は十分か
- 適切な命名になっているか
- 重複コードはないか
- 将来の変更に対応しやすいか

### 2. エラーケース
- 異常系の処理は考慮されているか
- 「この時こうなるけど対応してる？」の指摘
- バリデーション漏れはないか

### 3. 潜在的問題
- 「この実装だとこういう問題が起きるかも」の指摘
- パフォーマンス上の懸念
- セキュリティ上の懸念

### 4. 影響箇所
- メインに反映したらどこに影響するか
- 他の機能への副作用はないか
- 関連するAPIやコンポーネントの確認

### 5. ドキュメント整合性
- `issues/issue{番号}-xxx.md` が作成されているか（ローカルIssueファイル）
- backlog.md から完了した項目が削除されているか
- `issues/closed-issues.md` に追加されているか
- 関連READMEの更新が必要な場合、更新されているか

## 指摘への対応ルール

レビューで発見した問題は、重要度に応じて以下のように対応する。

### 致命的な指摘（必ず対応）
- セキュリティ上の問題
- データ破損の可能性
- 主要機能が動作しない

**対応**: その場で修正してからコミット

### 重要な指摘（対応推奨）
- エラーハンドリングの不備
- 明らかな設計上の問題

**対応**: その場で修正

### 軽微な指摘（対応任意）
- コードスタイル
- 将来的な改善提案

**対応**: その場で修正、またはユーザー判断で棄却

## レビュー記録テンプレート

```markdown
# レビュー: Issue #{番号} - {タイトル}

**レビュー日**: YYYY-MM-DD
**対象作業ログ**: dev-logs/YYYY-MM-DD_xxx.md
**GitHub Issue**: URL

---

## 変更概要

- 変更ファイル一覧
- 変更内容の要約

## レビュー結果

### 保守性・コード品質
- [ ] 指摘事項1
- [ ] 指摘事項2

### エラーケース
- [ ] 指摘事項

### 潜在的問題
- [ ] 指摘事項

### 影響箇所
- 影響を受けるファイル/機能

### ドキュメント整合性
- [ ] issues/issue{番号}-xxx.md 作成済み
- [ ] backlog.md から完了項目を削除済み
- [ ] closed-issues.md に追加済み
- [ ] 関連README更新済み（該当する場合）

## 対応状況

| 指摘 | 対応 | 完了 |
|------|------|------|
| xxx | 修正した / 仕様として許容 | - [ ] |

## 結論

- [ ] レビュー完了・コミット可
```

## 作業ログとの関係

| ファイル | 役割 |
|----------|------|
| dev-logs/YYYY-MM-DD_xxx.md | 何をやったか（実装内容の記録） |
| reviews/issue{番号}-review.md | 問題ないか（品質確認の記録） |

両方を参照することで、**何を作って、それが問題ないことを確認した**という完全な記録になる。
