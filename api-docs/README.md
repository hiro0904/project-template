# APIドキュメント

**最終更新**: YYYY-MM-DD

API仕様書を管理するフォルダです。

## 方針: 実装ファースト・検証ベース

**重要**: ドキュメントは実際のAPI動作を検証してから作成・更新する。

### 禁止事項
- ❌ 実装を確認せずに仕様書を更新する
- ❌ 想像でリクエスト・レスポンス例を作成する

### 必須事項
- ✅ 実際のAPI実行テスト後にドキュメント更新
- ✅ エラーや未実装状況を明確に記録

---

## ドキュメント作成手順

### 1. API実装完了後

```bash
# サーバー起動確認
npm run dev  # または該当コマンド

# 実行テスト
curl -s "http://localhost:3001/api/xxx" | jq .
```

### 2. 実測データでドキュメント作成

実際のレスポンスをコピーして仕様書に記載。

### 3. エラーケースも検証

正常系だけでなく、異常系も確認して記載。

---

## ファイル構成

```
api-docs/
├── README.md              # 本ファイル
├── api-spec-template.md   # API仕様書テンプレート
└── {domain}/              # ドメイン別フォルダ（任意）
    └── api-specification.md
```

---

## API仕様書テンプレート

各APIは以下の形式で記載:

```markdown
## {HTTPメソッド} {エンドポイント}

### 概要
{このAPIが何をするか}

### リクエスト

**パラメータ**:
| 名前 | 型 | 必須 | 説明 |
|------|-----|------|------|
| ... | ... | ... | ... |

**ボディ**（POST/PUT/PATCHの場合）:
```json
{
  "field": "value"
}
```

### レスポンス

**成功時** (200):
```json
{
  "実際のレスポンス": "をここに記載"
}
```

**エラー時** (4xx/5xx):
```json
{
  "error": "エラーメッセージ"
}
```

### 実行例

```bash
curl -X GET "http://localhost:3001/api/xxx"
```
```

---

## 更新タイミング

- API実装完了時
- API仕様変更時
- バグ修正でレスポンス形式が変わった時
